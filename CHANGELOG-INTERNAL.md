# Changelog

Tutte le modifiche rilevanti alla **spec OpenAPI** (`openapi/magware.yaml`) sono documentate in questo file.

Il formato è basato su [Keep a Changelog 1.1.0](https://keepachangelog.com/it/1.1.0/) e il progetto aderisce al [Semantic Versioning](https://semver.org/lang/it/), allineato al campo `info.version` della spec.

## Scope

Questo changelog documenta **solo le modifiche al contratto pubblico delle API** — cioè cambi visibili a chi consuma `openapi/magware.yaml` o naviga la reference su `api.magware.it`: nuovi endpoint, modifiche a schema/parametri/response, deprecation, breaking change, fix di esempi o testi user-facing.

I cambi a tooling, configurazione del repo, CI, `CLAUDE.md` e altre attività interne **non rientrano** in questo file: restano tracciati nel `git log` e, per le scelte strutturali, nella sezione "Storia delle decisioni" di `CLAUDE.md`.

## [Unreleased]

> Tutto il lavoro fatto dall'inizio del repo è in `[Unreleased]` finché non si rilascia la prima release pubblica (al termine della revisione della spec). A quel punto l'intero contenuto qui sotto verrà consolidato in inglese in `CHANGELOG.md`, taggato con il primo `vX.Y.Z` e promosso a sezione versionata anche qui.

### Added

- `item_stock` schema: aggiunti campi `variant_quantity` (integer, opzionale) e `variant_reserved_quantity` (integer, opzionale) — emessi sempre da `GET /stocks` e `GET /stocks/{item_code}` via `json.setint` su colonne `min_tot` e `min_pre` (aggregati per variante su tutti i lotti e stock type). Opzionali per coerenza con `reserved_quantity` (schema condiviso con `adjustment.items`). Sorgente: `n_get_stocks.sru`.
- `item_stock` schema: aggiunto campo `reserved_quantity` (number, decimal, opzionale) — emesso sempre da `GET /stocks` e `GET /stocks/{item_code}` via `json.setdecimal` su colonna `qta_pre` (sorgente `n_get_stocks.sru`). Campo opzionale (non `required`) perché lo schema è condiviso con `adjustment.items`, dove il backend degli adjustment non emette questo campo. Esempi nei due endpoint `/stocks` aggiornati con valori realistici.

- Importata la spec attuale di Magware API da Stoplight (bundled reference, 20 endpoint, 8 tag, 18 schema). Sostituito integralmente lo skeleton placeholder che era in `openapi/magware.yaml`.
- Aggiunto il tag globale `Delivery Notes` (era usato da 2 operation ma mancava nella dichiarazione `tags:` globale).
- Aggiunte description significative a tutti i 7 tag (prima erano placeholder identici al nome).
- Aggiunta nota su `info.description` che chiarisce: la reference documenta il sandbox pubblico (`https://sandbox.magware.it:9999`); gli URL di produzione sono per-tenant e comunicati da Restore in fase di onboarding.
- `POST /items` ora documenta esplicitamente la **bulk creation**: il request body accetta `oneOf` un singolo item oppure un array di items (min 1, max 100). Allineamento al backend reale (`magware-refs/n_post_items.sru` linee 258-300) che già supportava entrambi i formati ma la spec esponeva solo il singolo. Aggiunti esempi sia `single` sia `bulk`.
- `POST /items` response: aggiunta `200 OK` per il caso bulk con schema dedicato (array di `bulk_item_result` — un'entry per item input con `position`, `item_code`, `status`, `description`). La `201 Created` resta per il caso singolo. Il backend ritorna `200` anche se alcuni item nell'array hanno fallito; l'integratore deve scorrere l'array e leggere `status` per ogni entry. Comportamento documentato nella `description` dell'operation.
- Nuovi schemi in `components.schemas`: `success` (envelope per response 2xx single-resource, prima inline-only in `components.responses.success`) e `bulk_item_result` (outcome per item nella bulk creation).

- L'header `owner` (tenant code) è stato promosso da parameter a `securityScheme`. Prima era dichiarato in `components.parameters.owner` e referenziato 22 volte nelle operation; ora è uno scheme di tipo `apiKey` accanto a `APIKey`, applicato globalmente via il blocco `security: [{ APIKey: [], Owner: [] }]` (AND, entrambi obbligatori). Motivo: una `APIKey` è legata 1:1 a un `owner`, quindi sono semanticamente una credenziale composita; la nuova forma è più ergonomica nella reference (pannello "Authorize" unico) e nei client SDK generati (configurati una volta, niente parameter ripetuto). **Zero impatto wire**: gli header HTTP inviati dai client restano `APIKey: ...` + `owner: ...`, esattamente come prima. Possibile impatto solo per chi rigenera SDK dalla spec (struttura diversa nel codice generato, comportamento HTTP identico).

- **Track 4 — Default / nullable / required (rimozione default obsoleti + required allineati)**. Audit sistematico dei 31 `default:` presenti nella spec (verificati uno per uno contro i sorgenti PB in `magware-refs/` e contro il DDL `mw21r2.sql`). Identificati 12 default da rimuovere, raggruppati in 5 cause distinte, più un `required` allineato al backend:
  - **Default su schema response (3)**: `delivery_prepared_details.cancelled.default: false`, `delivery_prepared_details.channel.default: WS`, `delivery_cancelled_details.channel.default: WS`. Un `default:` su una response è semanticamente vuoto (il server o emette il valore o omette il campo, il client non "completa" un default su qualcosa che riceve). Per `channel` il backend popola sempre il campo da `ds_del.getitem(1, "source")` (`n_get_deliveries_prepared.sru:112`, `n_get_deliveries_cancelled.sru:32`); `cancelled` non viene neanche emesso (campo già `deprecated`).
  - **Example accidentale finito come default (1)**: `delivery_creation.cash_on_delivery.value.default: 123.45`. Nessun backend può "defaultare" un valore di contrassegno a una cifra inventata. Il backend (`n_post_delivery.sru:82`) salva pari pari ciò che riceve; il valore resta `required` dentro l'oggetto `cash_on_delivery`.
  - **Default in italiano in spec inglese (4)**: `delivery_creation.destination.formal_salutation.default: "Egr. Sig."`, `delivery_creation.destination.informal_salutation.default: "Caro"`, e i due omologhi su `delivery_creation.billing`. Il backend (`n_post_delivery.sru:223-224, 276-277`) usa `checkandset(cli, ..., json.getstring(...))` — non inietta nulla se il campo è assente, salva la stringa vuota. I default erano residui di vecchi example italiani.
  - **Default non applicati dal backend (2)**: `item.variants[].quantity.default: 1` e `item.variants[].quantity_per_package.default: 1`. Il backend non ha un branch `else` che inietti questi valori se mancanti (verificato in `n_post_items.sru:185-186, 220-222` e `n_put_items.sru:85-88, 112-114`). I 2 default in spec erano informazione fuorviante.
  - **Default ridondanti su campi già `required` (2)**: `delivery_creation.cash_on_delivery.currency.default: "EUR"` e `shipment_creation.cash_on_delivery.currency.default: "EUR"`. Dentro l'oggetto `cash_on_delivery` la `currency` è già `required: [currency, value]`, quindi il `default:` non si applica mai a livello OpenAPI (il campo non può mancare se l'oggetto è presente). Il backend non inietta alcun default DB (`richiesta.contrassegno_valuta` è `char(3) NULL` senza `DEFAULT` constraint, vedi `mw21r2.sql:344`). Riformulate anche le 2 description: da `'3 digit ISO 4217 Cash On Delivery currency (default: "EUR")'` a `'3-digit ISO 4217 Cash On Delivery currency. Conventional value is "EUR".'` (mantiene l'hint ma elimina il riferimento a un "default" che non esiste).
  - **Required esteso su variants**: `item.variants[].required` esteso da `[code, description]` a `[code, description, quantity]`. Il backend richiede di fatto `quantity` per ogni variant (`checkandset(min, "qta", ..., "gtzero")` in `n_post_items.sru:186`); senza il campo o con valore non > 0 emette `400`. La spec ora documenta il vincolo reale invece di suggerire un default fittizio per `quantity`.

  Restano in spec 19 default reali "validi", confermati allineati al backend (codice o DDL):
  - 8 boolean su `item.*` (`lot_control`, `serial_number_control`, `expiration_date_control`, `expedition_date_control`, `obsolete`, `variable_weight`, `variable_volume`, `can_be_added_to_deliveries`) — backend usa `BoolOf` (false se mancante); 3 confermati anche da DDL (`fl_peso_var`, `fl_volume_var`, `fl_can_be_added` con `DEFAULT ((0))`).
  - 5 boolean su `delivery_creation.*` (`do_not_pick`, `do_not_print`, `print_delivery`, `ship_on_demand`, `allow_extra_items`) — `print_delivery` e `allow_extra_items` confermati anche da DDL (`fl_stampa_bolla`, `fl_extra_items` con `DEFAULT ((0))`).
  - 2 default backend su `item.variants[].volume = 0.01` e `weight = 0.1` (iniezione esplicita in `n_post_items.sru:202,209` e `n_put_items.sru:96,101`).
  - 2 boolean su `item.variants[].shippable` e `item.variants[].default` (false se mancante).
  - `delivery_creation.priority: 0` — backend `if priority > 0 then set, else 0` (`n_post_delivery.sru:49-53`); confermato anche da DDL (`richiesta.priorita DEFAULT ((0))`).
  - `delivery_creation.partial_allowed: true` — il backend setta `fl_parziale = 1` come default implicito a inizio creazione richiesta, sovrascritto solo se `partial_allowed` è esplicitamente presente nel JSON (`n_post_delivery.sru:66-70` + iniezione implicita confermata da Carlo). La colonna DB `richiesta.fl_parziale` è `tinyint NOT NULL` senza `DEFAULT` constraint: il default viene applicato lato code, non lato DB.

  **Default impliciti backend valutati ma deliberatamente non documentati nella spec**:
  - `high_priority`: il backend implicitamente setta `fl_urgente = 0` e lo sovrascrive a 1 solo se `high_priority = true` nel JSON. Il default API effettivo sarebbe `false`, ma il campo è già `deprecated` (sostituito da `priority`); non aggiunto `default: false` per non sovra-documentare un campo deprecato.
  - `freight_term`: il backend implicitamente setta `porto = "Franco"` a inizio creazione richiesta, ma poi `checkandset(rich, "porto", json.getstring("freight_term"))` (`n_post_delivery.sru:78`) **sovrascrive incondizionatamente**: la definizione di `checkandset` (`n_impexp.sru:321-384,394-400`) mostra che la variante 3-parametri applica solo un `validlen` check (lunghezza max) e poi chiama sempre `pdo.set(attr, value)`. Quando `freight_term` è assente nel JSON, `getstring` restituisce `""`, `validlen` passa (0 ≤ maxlen), e `porto` viene sovrascritto con `""`. Il "Franco" del costruttore si applica solo a flussi non-API (es. UI con form precompilato), non all'integrazione REST. Conseguenza: il default API effettivo è la stringa vuota, **non** "Franco" → non aggiunto `default: "Franco"` in spec (sarebbe documentazione fuorviante).
  - **Asimmetria backend osservata** (`Known issue` candidata per CHANGELOG-INTERNAL futuro o discussione con backend team): per `cash_on_delivery.currency` su `POST /shipments` il dev ha usato `coalesce(json.getstring("cash_on_delivery.currency"), "EUR")` (`n_post_shipments.sru:121`) che effettivamente difende il default "EUR" anche dall'override con stringa vuota. Lo stesso pattern non è applicato a `porto` su `POST /deliveries`: o l'intento era "porto è opzionale, vuoto se non specificato" (e allora il `set("porto","Franco")` del costruttore è dead code per le chiamate API), oppure è una mancanza di coerenza — da chiarire con backend team.

- **Track 4bis — maxLength audit globale** (verificato property-by-property contro DDL `magware-refs/mw21r2.sql` + mapping JSON→tabelle ricavato dai sorgenti PB). ### Known issues

- L'esempio `Prepared delivery` di `GET /delivery_notes/{id}` ha al root campi NON dichiarati nello schema `delivery_note_details`: `preparation_date`, `cancelled`, `channel`, `delivery_note` (sotto-oggetto). Fino al fix backend, gli integratori che ricevono un 404 troveranno nel body una stringa non-JSON (es. `"Item with code '1234567890AB' not found for owner 'RST'"`) anziché l'envelope JSON.
